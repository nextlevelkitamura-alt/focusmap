# Focusmap UI Constitution

## Brand Invariants

- Focusmapは黒基調の高密度な操作ダッシュボードであり、マーケティングLP風、装飾カード中心、単一色グラデーション中心の見た目へ寄せない。
- 色、状態色、角丸、線幅、アイコン、文字密度は既存の `docs/CONTEXT.md` と共通UIコンポーネントを優先する。
- アイコンは原則 `lucide-react` を使い、stroke、サイズ、色の扱いを揃える。
- テーマ変更は目的ではない。修正は「壊れたUIをFocusmapらしい完成度へ戻す」ために行う。
- 画面ごとに別アプリのような配色、フォント、余白、ボタン形状を持ち込まない。

## Desktop App Principles

- デスクトップは俯瞰と詳細編集の同時性を優先する。全画面を覆う編集フォームより、右サイドバー、インスペクタ、ポップオーバー、スプリットビューを優先する。
- 一覧、カレンダー、マップなどの主対象は表示したまま、選択中の詳細だけを横に出す。
- 小さな値変更、カレンダー選択、通知時間、表示設定はポップオーバーやドロップダウンにする。削除確認や危険操作だけダイアログにする。
- デスクトップでモバイルのボトムシートを横幅いっぱいに引き伸ばさない。編集領域は必要な情報量に応じておおむね360-440px、広いフォームでも主画面の文脈を残す。
- 横幅を使う理由がないフォーム、1行だけのために画面幅を占有するセレクト、巨大な空白、中央に薄く浮く単独フォームは避ける。
- カレンダーは「時間軸を見ながら編集」を守る。予定クリック後は、背景の予定配置を暗くしすぎず、右側または近接ポップオーバーで編集できる状態を基本にする。

## Mobile App Principles

- モバイルは片手操作と一画面一目的を優先する。PCの多列レイアウトをそのまま縮小しない。
- 主要ナビは `Todo` / `メモ` / `マップ` / `チャット` / `設定` の下部ナビを基本にする。
- タップターゲットは最低44px、下部操作はsafe areaを考慮し、キーボード表示時に入力や確定ボタンが隠れないようにする。
- 詳細編集はボトムシート、ドリルイン画面、段階開示を使う。大量の項目を1枚のシートに押し込まない。
- 重要な状態やエラーは操作地点の近くに出す。成功するまで無反応に見えるUIは避け、楽観的UIと保存中表示を使う。
- 初回ロードや復帰時に白画面を出さない。既存の暗色起動シェル、キャッシュ表示、細い進捗バーの方針を守る。

## Cross-Platform Consistency

- 同じ概念には同じ名前、同じアイコン、同じ状態色を使う。
- プラットフォーム差は配置、密度、入力方式、safe area、画面分割で吸収する。ブランドの視覚言語は共通にする。
- デスクトップとモバイルで同じ機能を持つ場合も、見た目を完全一致させるのではなく、同じ情報構造と状態表現を維持する。
- 例: 予定編集は、デスクトップでは右インスペクタ、モバイルではボトムシートでよい。ただし項目名、状態、保存/削除/通知/カレンダー選択の意味は同じにする。

## Required Interaction Standards

- 追加、貼り付け、削除、並び替え、状態変更は原則として楽観的UIにする。
- 保存中は低透明度、仮表示、ローディング、未確定ラベルなどで処理中だと分かる状態を出す。
- API失敗時だけ元の状態へ戻し、操作地点の近くにエラーを出す。
- テキストは親要素内に収める。ボタン、カード、チップ、予定カード内の長い文言は折り返し、省略、行数制限を明確に決める。
- UI部品の寸法はhover、focus、loading、長文、件数増加で揺れないようにする。
- 状態を色だけで伝えない。ラベル、アイコン、位置、枠線、説明を併用する。

## P0 Anti-Patterns

- 白画面、クライアント例外、Hydration破綻、画面遷移不能。
- モバイルで主要操作がsafe areaやキーボードに隠れる。
- デスクトップで主画面を巨大フォームが覆い、文脈を見ながら編集できない。
- 画面やタブごとに別アプリのようなテーマ、フォント、角丸、アイコンが出る。
- 重要な操作の成功/失敗がUIに反映されない。
- 横幅や縦幅を過剰に占有し、情報密度がFocusmapの操作面として成立しない。

## 95-Point Proposal Shape

改善案は必ず次の形にする。

1. 何が壊れているか。
2. Focusmapらしさとして守るものは何か。
3. 95点以上の理想状態は何か。
4. デスクトップとモバイルでどう分けるか。
5. どのコンポーネント、状態、レイアウトを変えるか。
6. 触らないファイル、壊してはいけない既存挙動は何か。
7. P0/P1/P2の完了条件は何か。

点数は最後の診断でよい。点数だけ、または抽象的な「洗練させる」で終わらせない。
