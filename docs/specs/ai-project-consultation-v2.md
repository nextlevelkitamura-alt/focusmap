# AIプロジェクト相談 v2 — 設計仕様書

## 概要
AIプロジェクト相談を、対話型のプロジェクト理解 + マインドマップ操作 + 自動要約の統合機能にアップグレードする。

## 目的
- フリーテキストで柔軟にプロジェクトについて相談できる
- 会話の中からAIが「マインドマップ操作」「予定追加」を判断し提案する
- プロジェクトの概要を自動で要約・保存し、常に最新に保つ

---

## ユーザー体験フロー

### 1. 開始（ハイブリッド型）
- 選択肢: 「構造を整理」「タスク追加」「自由に相談」
- **またはフリーテキストでそのまま話し始められる**

### 2. プロジェクト特定 → コンテキスト読み込み
```
[会話中にプロジェクトが特定される]
  ├─ 要約あり → ai_context_documents から読み込み、AIの理解を深める
  └─ 要約なし → AIが自然にプロジェクト概要をヒアリング
```

### 3. 対話フェーズ
- フリーテキストで自由に会話
- AIはプロジェクト要約 + マインドマップ構造（ツリーテキスト）を把握

### 4. 提案 → 承認フェーズ
AIが振り分け判断し、**毎回ユーザーに確認**する：
- 「"〇〇機能" をマインドマップに追加しますか？」→ 承認ボタン
- 「"△△" をマインドマップから削除しますか？」→ 承認ボタン
- 「"□□" を予定に入れますか？」→ 承認ボタン

### 5. プロジェクト要約の自動更新
- 会話の中で新しい情報が出たら、スピナー + 「会話を要約しています」表示
- 既存要約 + 新情報 → **再要約して上書き保存**（追記ではない）
- 常に1つのコンパクトな最新版を維持（200〜300字）

---

## 振り分けロジック

| 判断基準 | → マインドマップ | → 予定/タスク |
|---------|----------------|-------------|
| 性質 | 構造的・長期的 | 短期的・一時的 |
| 例 | 「認証機能が必要」 | 「明日までにログイン画面のモック作る」 |
| 重要度 | プロジェクトの柱となるもの | すぐやるべき作業 |

---

## AIに送るコンテキスト（コスト最適化）

### 送るもの
| データ | 形式 | 推定トークン |
|-------|------|------------|
| プロジェクト要約 | ai_context_documents のテキスト | 少（200〜300字） |
| マインドマップ構造 | ツリー形式テキスト（ノード名+親子関係のみ） | 少〜中 |
| プロジェクト一覧 | 名前+IDのリスト | 少 |

### 送らないもの
- ノードに付随する画像
- ノードの詳細メモ
- 他プロジェクトの詳細

### マインドマップ構造のフォーマット例
```
プロジェクト: しくみかアプリ
├─ 認証機能 [group:uuid1]
│  ├─ Googleログイン ✅ [task:uuid2]
│  └─ メールログイン [task:uuid3]
├─ ダッシュボード [group:uuid4]
│  ├─ マインドマップ表示 ✅ [task:uuid5]
│  └─ カレンダー連携 [task:uuid6]
└─ AI機能 [group:uuid7]
   ├─ チャット ✅ [task:uuid8]
   └─ プロジェクト相談 [task:uuid9]
```
※ IDを付与することで、AIが削除対象を正確に特定できる

## 新アクションタイプ

### `add_mindmap_group`
マインドマップにグループ（ルートノード）を追加
```json
{
  "type": "add_mindmap_group",
  "params": { "title": "新機能名", "project_id": "xxx" },
  "description": "マインドマップに「新機能名」グループを追加します"
}
```

### `add_mindmap_task`
マインドマップにタスクを追加（親ノード指定）
```json
{
  "type": "add_mindmap_task",
  "params": { "title": "タスク名", "parent_id": "group-uuid", "project_id": "xxx" },
  "description": "「親グループ」に「タスク名」を追加します"
}
```

### `delete_mindmap_node`
マインドマップからノードを削除
```json
{
  "type": "delete_mindmap_node",
  "params": { "node_id": "uuid", "node_title": "表示用タイトル" },
  "description": "マインドマップから「表示用タイトル」を削除します"
}
```

### `add_calendar_event`（既存）
予定をカレンダーに追加（変更なし）

## プロジェクト要約の再要約ロジック

### 保存先
`ai_context_documents` テーブル（プロジェクトフォルダ配下）

### フロー
```
1. 既存の要約テキストを読み込み
2. 現在の会話内容と統合
3. AIに再要約を依頼（200〜300字に圧縮）
4. 上書き保存
5. UIにスピナー + 「会話を要約しています」ログ表示
```

### AIへの再要約指示
```
以下のプロジェクト概要と新しい会話内容を統合し、
200〜300字で最新のプロジェクト概要として再要約してください。
古い情報は新しい情報で更新してください。

【既存の概要】
{existing_summary}

【新しい会話の内容】
{new_conversation_highlights}
```

## 実装ステップ

### Step 1: マインドマップ構造テキスト生成
- `src/lib/ai/context/mindmap-context.ts` を新規作成
- groups/tasks からツリー形式テキストを生成する関数

### Step 2: プロジェクト相談プロンプト改修
- `project-consultation.ts` を改修
- 振り分けロジック・ヒアリング・再要約の指示を追加
- マインドマップ構造をコンテキストに含める

### Step 3: chat route 改修
- マインドマップ構造をAPIで取得してプロンプトに注入
- プロジェクト特定時の動的コンテキスト読み込み
- `project_context_update` を再要約型（上書き）に変更

### Step 4: マインドマップ操作エンドポイント
- `src/app/api/ai/chat/execute/route.ts` に新アクション追加
- `add_mindmap_group`, `add_mindmap_task`, `delete_mindmap_node`

### Step 5: UI改善
- 承認カード（マインドマップ追加/削除用）
- 要約中スピナー + ログ表示
- ハイブリッド型の開始画面

### Step 6: 再要約ロジック
- 既存要約 + 新情報 → 再要約 → 上書き保存
- `ai_context_documents` への保存処理
