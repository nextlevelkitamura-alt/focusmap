# Research Workflow

## Goal

実装前に、Focusmapらしさを維持したまま95点以上へ上げるためのUI調査を行う。
調査subagentはreadonlyにする。

## Subagent Roles

### Desktop Benchmark

- Apple、Google、YouTube、Notion級の成熟デスクトップUIから、Focusmapへ転用できる設計原則を抽出する。
- 多列レイアウト、右インスペクタ、ポップオーバー、サイドパネル、状態表示、密度を重点的に見る。
- 「デスクトップでやってはいけないモバイル流用」を具体化する。

### Mobile Benchmark

- 片手操作、下部ナビ、ボトムシート、safe area、44pxタップ、復帰時ローディングを重点的に見る。
- 同じFocusmapに見せながら、PCと違う情報密度にする方法を出す。
- 「モバイルでやってはいけないPC多段UI」を具体化する。

### Current Focusmap Inventory

- 対象コードと `docs/CONTEXT.md` を読み、現在の色、角丸、線幅、アイコン、密度、既存コンポーネントを整理する。
- 壊れている画面だけでなく、同じアプリらしさの基準になる良い既存UIも拾う。

### QA / Test Review Design

- P0/P1/P2の確認観点を作る。
- ユーザーが明示した場合だけ実行するテスト/表示確認候補を出す。
- 目視確認だけでなく、長文、件数増加、狭幅/広幅、空状態、保存失敗、ローディングを含める。

## Research Rules

- まずFocusmapの既存仕様を読む。外部UIの真似を先にしない。
- 外部調査を行う場合は、公式ガイドや成熟プロダクトの実UIから原則だけ抽出する。
- 調査結果は「一般論」ではなく、対象画面に適用した95点案に変換する。
- 点数だけで終わらせない。

## Output

1. Desktopで守るべき原則
2. Mobileで守るべき原則
3. Focusmapの既存テーマとして維持するもの
4. 対象画面のP0/P1/P2
5. 95点以上の理想状態
6. 実装分解に渡す制約
