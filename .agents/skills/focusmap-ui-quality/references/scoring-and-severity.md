# Scoring And Severity

## Score Is Diagnostic

点数は意思決定の補助であり、成果物ではない。
95点未満の評価を出す場合は、95点以上へ上げるための具体的な修正案も同時に出す。

## Severity

- P0: 画面が使えない。白画面、例外、主要操作不能、保存/削除の重大破綻、モバイルで操作が隠れる、認証/データ喪失リスク。
- P1: 実用上かなり悪い。デスクトップで文脈を見ながら編集できない、モバイル/PCで別アプリに見える、主要導線が遠い、重要状態が伝わらない、長文でUIが崩れる。
- P2: 品質低下。余白過多、密度不足、状態色の弱さ、ラベルの曖昧さ、hover/focus不足、微妙な不整列。
- P3: 仕上げ。コピー、微調整、軽いアニメーション、優先度の低い見た目の磨き込み。

P0は即修正対象。P1は完了扱いにしない。P2は残す場合に理由と次回対応を明記する。

## 100-Point Rubric

- 20: 機能成立。主要操作が成功し、失敗時も戻せる。
- 15: 情報設計。主目的、補助情報、危険操作、詳細編集の優先順位が明確。
- 15: プラットフォーム適合。デスクトップは俯瞰+詳細、モバイルは片手+段階開示。
- 15: Focusmap視覚統一。色、角丸、線幅、lucide、密度、状態表現が既存と揃う。
- 10: レスポンシブ/可変状態。長文、件数増加、狭幅、広幅、ローディング、空状態で崩れない。
- 10: 操作フィードバック。楽観的UI、保存中、エラー、完了、disabled理由が分かる。
- 10: アクセシビリティ。44pxタップ、フォーカス、コントラスト、キーボード操作、色だけに依存しない状態。
- 5: 仕上げ。微細な整列、ラベル、アニメーション、不要な装飾の削減。

## Review Output

レビュー結果は次の順で出す。

1. P0/P1 findings。ファイルと行、または画面領域を具体化する。
2. 95点以上へ持っていく修正案。実装方針、コンポーネント、状態、レイアウト単位で書く。
3. 既存テーマのうち守るべき点。
4. 実装時に触ってよい範囲と触らない範囲。
5. 必要なら点数。点数だけで終わらせない。
6. ユーザーが明示した場合だけ、実行すべき確認コマンドや表示確認手順。

## Pass Gate

- P0が0件。
- P1が0件、またはユーザーが明示的に延期を承認している。
- DesktopとMobileの体験差が説明できる。
- Focusmapの既存テーマを変えていない。
- 実装方針が「95点以上の理想状態」まで具体化されている。
