# 設計プラン: モバイルUI再設計 + 今日ビュー + 習慣機能

## 目標

スマホファーストのUI体験を構築し、「今日やるべきこと」に集中できる画面と習慣トラッキングを統合。
PC でも同等の機能をアクセスできるレスポンシブ設計。

---

## 現状分析

### 現在のボトムナビ（5タブ）
```
[ 今日 ] [ 俯瞰 ] [ 追加 ] [ 集中 ] [ 設定 ]
```
- 「俯瞰」「追加」「集中」のページは**未実装**（bottom-nav.tsx にルート定義のみ）
- スマホでは CenterPane（マインドマップ + タスクリスト）がフル幅表示
- 左サイドバー・右サイドバー（カレンダー）はスマホで**非表示**

### 問題点
1. スマホで「カレンダー」にアクセスできない
2. 「今日やること」に特化したビューがない
3. タイマー機能がタスクリスト内に埋もれている
4. 習慣トラッキング機能がない
5. ボトムナビの半分が未実装ルート

---

## 新しいボトムナビ設計

### 構成（4タブ）
```
[ 今日 ] [ マップ ] [ 習慣 ] [ 設定 ]
```

| タブ | ルート | アイコン | 説明 |
|------|--------|---------|------|
| 今日 | `/dashboard` | CalendarDays | 今日のスケジュール + タスク + タイマー |
| マップ | `/dashboard/map` | Network | マインドマップ（既存 CenterPane） |
| 習慣 | `/dashboard/habits` | Target | 習慣トラッカー |
| 設定 | `/dashboard/settings` | Settings | 設定（既存） |

### 設計判断
- **4タブに削減**: 5タブは指が届きにくい。4タブで親指操作に最適化
- **「追加」タブ廃止**: 各画面内の FAB（+ボタン）で対応
- **「集中」タブ廃止**: 「今日」ビュー内のタイマー機能に統合
- **「俯瞰」→「マップ」**: より直感的な名称に変更

#### 構成要素

**A. 習慣バー（上部固定）**
- 今日の習慣を横一列で表示
- 各習慣はタップで完了/未完了を切り替え
- 週の達成状況を小さなドット（7つ）で表示
- スワイプで前後の日を確認可能

**B. タイムラインビュー（メイン）**
- Google Calendar イベント + スケジュール済みタスクを時間軸で統合表示
- 現在時刻のインジケーター（赤線）
- 空き時間スロットの可視化
- タスクカードにタイマーボタン（▶）を直接配置

**C. 未スケジュールタスクセクション（下部）**
- 今日期限だがまだ時間未割り当てのタスク
- ドラッグ&ドロップでタイムラインに配置可能
- タップでクイック時間設定

**D. ドラッグ&ドロップ**
- 未スケジュールタスクをタイムラインにドラッグ → 時間割り当て
- タイムライン内のタスクをドラッグ → 時間変更
- 右端のカレンダーとの D&D は既存機能を活用

## データベース設計

### 新規テーブル: `habits`

```sql
CREATE TABLE habits (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id UUID NOT NULL REFERENCES auth.users(id) ON DELETE CASCADE,
    title TEXT NOT NULL,
    icon TEXT,                          -- 絵文字アイコン（💧, 📖 等）
    frequency TEXT NOT NULL DEFAULT 'daily',  -- 'daily', 'weekdays', 'custom'
    frequency_days INTEGER[],          -- custom の場合: [1,2,3,4,5] (月-金)
    target_count INTEGER DEFAULT 1,    -- 1日の目標回数
    order_index INTEGER DEFAULT 0,
    is_active BOOLEAN DEFAULT true,
    created_at TIMESTAMPTZ DEFAULT NOW(),
    updated_at TIMESTAMPTZ DEFAULT NOW()
);

-- RLS
ALTER TABLE habits ENABLE ROW LEVEL SECURITY;
CREATE POLICY "Users manage own habits"
    ON habits FOR ALL USING (auth.uid() = user_id);
```

### 新規テーブル: `habit_logs`

```sql
CREATE TABLE habit_logs (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    habit_id UUID NOT NULL REFERENCES habits(id) ON DELETE CASCADE,
    user_id UUID NOT NULL REFERENCES auth.users(id) ON DELETE CASCADE,
    completed_date DATE NOT NULL,       -- 完了日（時間は不要）
    count INTEGER DEFAULT 1,            -- 回数（複数回の場合）
    created_at TIMESTAMPTZ DEFAULT NOW(),

    UNIQUE(habit_id, completed_date)    -- 1日1レコード
);

-- RLS
ALTER TABLE habit_logs ENABLE ROW LEVEL SECURITY;
CREATE POLICY "Users manage own habit logs"
    ON habit_logs FOR ALL USING (auth.uid() = user_id);

-- インデックス
CREATE INDEX idx_habit_logs_date ON habit_logs(habit_id, completed_date);
CREATE INDEX idx_habit_logs_user_date ON habit_logs(user_id, completed_date);
```

## API 設計

### 習慣 CRUD
```
GET    /api/habits              → 一覧取得
POST   /api/habits              → 新規作成
PATCH  /api/habits/[id]         → 更新
DELETE /api/habits/[id]         → 削除
```

### 習慣ログ
```
POST   /api/habits/[id]/log     → 完了記録（toggle）
GET    /api/habits/logs?from=&to=  → 期間指定でログ取得
```

### 今日ビュー用
```
GET    /api/today               → 今日のタスク + イベント + 習慣を統合取得
```

## 推奨実装順序

→ **Phase 1 から /impl で着手**

Phase 1 でルーティング基盤を整え、画面遷移が動く状態を先に作る。
その後、各ビューの中身を順次実装していく。
